# LinguaLens — Real-time English Conversation Coach for Even G2

OpenAI Build Week 2026 提出作品（カテゴリ: Education）。締切: 2026-07-21 17:00 PDT (= 07-22 09:00 JST)。

## プロダクト定義（1文）

英会話中、Even G2 スマートグラスの HUD に「答えではなく思考を促すヒント」をそっと出す、装着型の英語会話コーチ。

## 誰の・何の課題か（当事者性）

日本人英語学習者（作者本人）は、実会話の最中に「言いたいことが出てこない」「相手の単語が聞き取れない」瞬間に学習機会を失う。スマホを見れば会話が壊れる。グラスの HUD なら視線を外さずに補助を受けられ、しかも答えを丸写しさせず「想起の手がかり」だけを出せば学習効果が残る（Belsky の "not an answer machine" 思想に整合）。

## プライマリユースケース（2026-07-19 追加・オーナー実体験）

作者は日常的に ChatGPT の GPT-Live（全二重音声）と英会話練習をしており、言い回しに詰まる瞬間が頻発する。LinguaLens は G2 マイクで「部屋の会話」を聞く設計のため、**相手が人間でも GPT-Live でも無改修で動作する**。GPT-Live 練習が主戦場、実会話が本番、という二段構え。運用: ChatGPT Voice をバックグラウンド再生にし Even アプリを前面に置く（プラグインは foreground 限定のため）。デモ動画のコアシーンはこの「GPT-Live と会話しながら HUD にヒントが出る」実機映像を使う。

## コア体験（1機能を完璧に）

会話モード（唯一のモード）:
1. G2 のマイクで会話音声を継続キャプチャ（bridge.audioControl → 16kHz PCM mono 16bit）
2. OpenAI ASR でリアルタイム文字起こし（自分と相手の英語）
3. GPT-5.6 が「コーチング介入判定」を実行し、必要な時だけ HUD に短いカードを出す:
   - **HINT**: 言い淀み・日本語混じりを検知 → **すぐ口に出せる完全な英文を1〜3択で提示、各選択肢に母国語（日本語）の短い意味ラベルを併記**（英文は各6語以内、日本語ラベルは3〜8文字。文脈の確信度が高ければ1文、曖昧なら最大3択。番号付き表示）。※2026-07-19 オーナー決定・変更不可: 当初の「最初の2語のみ」案は破棄。in-the-moment は実用性優先（完全文+母国語ラベル）、学習効果は RECAP と次回復習で担保する設計に変更。母国語は当面日本語固定、将来は設定切替
   - **WORD**: 相手の発話中の高難度語を検知 → 語 + 3語以内の平易な言い換え
   - **RECAP**: 会話の切れ目で、直近のやり取りから「今言えなかった表現1つ」を復習カードとして提示
4. セッション終了時（ダブルタップ）、学習ログを localStorage に保存し、次回起動時に前回の復習カードを1枚出す

静かさが正義: HUD 表示は最大2秒で読める分量。介入しすぎないこと自体が製品価値。GPT-5.6 に「介入しない」判定を明示的に許す。

## アーキテクチャ

```
[G2 glasses] --BLE--> [Even App WebView: 本アプリ]
                         ├─ AudioPipeline: PCM chunk → OpenAI transcription API
                         ├─ CoachEngine: transcript window → GPT-5.6 (structured output)
                         │    intervention: {type: HINT|WORD|RECAP|NONE, text, ttl}
                         └─ HudRenderer: container 更新（下記 BLE 予算）
[Desktop] evenhub-simulator = ミラービュー（デモ動画・審査員試用の主画面）
```

- SDK: `@evenrealities/even_hub_sdk`、テンプレートは `asr` ベース
- モデル方針（コスト最小・オーナー決定・変更不可）: 高頻度の介入判定は **gpt-5.6-luna**（GPT-5.6 ファミリーの最安モデル。公式カタログ確認済み: sol=フラッグシップ(alias gpt-5.6)/terra=中間/luna=コスト最適）、低頻度の RECAP 生成のみ **gpt-5.6**（=sol、Build Week の「GPT-5.6 で構築」要件を満たすため最低1箇所は本体を使う）。ASR は **gpt-4o-mini-transcribe**（実在確認済み）。モデル名はすべて src/models.ts に一元管理し、コードに散らさない
- API キー: 実行時に `OPENAI_API_KEY` を設定画面（初回起動セットアップ）で入力し localStorage 保存（レビュー要件: 再プロンプト禁止）

## 設計改訂 v2（2026-07-20・実機テストの知見に基づく・変更不可）

### 判明した問題

1. **役割推定が壊れている**: G2 の4マイクアレイは SDK がマージ済み単一ストリームしか返さず（話者分離・DOA は公式ロードマップ上未実装）、音声から自分/相手を区別できない。現行実装は文字起こしのキーワード正規表現（"could you"→相手 / "I, we, uh"→自分）で推測しており頻繁に外れる。自分が "Could we push the deadline?" と言えば相手扱いになる。
2. **介入タイミングが不自然**: ASR チャンク（3.2秒）ごとに毎回判定を回すため、まだ話し終えていない途中で割り込む／逆に本当に詰まった瞬間を逃す。
3. **記録がない**: 直近 RECAP 1件しか永続化しておらず、チューニングの根拠が取れない（M5 テレメトリで対応中）。
4. 単価表が未確認（models.ts に TODO）。

### 解決方針

**原則: 「誰が話したか」を先に確定させようとせず、「会話が詰まったか」を検知して、役割判断はモデルに文脈ごと委ねる。**

1. **正規表現による役割推定を廃止**（`inferLiveRole` を削除）。ASR の結果は role 未確定のまま `utterance` として時刻・無音ギャップ付きで蓄積し、**役割の帰属は判定モデルに任せる**。プロンプトに「直近の発話列から、どれが学習者本人でどれが相手かを文脈から判断せよ。GPT-Live のような AI 相手の場合、相手の発話は流暢で長く、学習者の発話は言い淀みや日本語混じりを含む傾向がある」と明示。判定結果に `attributed_role` を含めて返させ、テレメトリに記録して精度を後から測れるようにする。

2. **毎チャンク判定をやめ、ストール検知でトリガーする**（コスト削減とタイミング改善を同時に達成）:
   - **フィラー検知**: "uh / um / えっと / あの / how do you say / なんだっけ" 等を含む発話が来たら即判定
   - **尻切れ + 沈黙**: 発話が接続詞・前置詞・冠詞等で終わっている（文が完結していない）状態で 1.2 秒以上の無音 → 判定
   - **相手の質問後の沈黙**: 直近発話が疑問形で終わり 2.5 秒以上の無音 → 判定
   - 上記いずれにも当たらない通常の会話進行では **判定 API を呼ばない**（WORD 用の難単語検知のみローカル語彙リストで前段フィルタし、ヒットした時だけ判定を回す）

3. **明示トリガー「タップで頼む」を追加**（確実な逃げ道）: テンプル/R1 の**シングルタップで即座にヒント要求**。ストール検知が外しても、ユーザーが能動的に助けを呼べる。誤爆ゼロの経路を持つことがデモでも効く。HUD には常時 "tap = help" を控えめに表示。

4. **単価表の是正**: 公式料金ページで gpt-5.6 系および gpt-4o-mini-transcribe の単価を確認して `PRICING` を実値に更新。TODO コメントを消す。

### 期待効果

- 誤爆の主因（役割誤判定 + 毎チャンク判定）を両方潰す
- API 呼び出し回数が大幅減 → コストは現行見積 $0.012/分 から数分の1に
- 「詰まった時だけ出る」という製品コンセプトが実装レベルで担保される

### ヒント品質の問題（2026-07-20 実機フィードバック・変更不可）

**症状**: 出てくる英文が「使い勝手が悪い」（汎用的すぎる／今言いたいことと違う／その場で口に出せない）。

**原因分析**:
1. **学習者の意図を示す最強のシグナルを捨てている**。日本人が詰まる時、多くは頭の中が日本語になり「えっと、締切延ばしたい…」のように**日本語の断片を漏らす**。これは意図そのものだが、現行プロンプトはこれを活用せず、英語の文脈からのみ推測している。
2. **文の途中から続けられない**。学習者が "The login flow. I am not sure how to..." まで言った状態で、無関係な別の文を3つ出しても口に出せない。**すでに言いかけた文の続き**でなければ使えない。
3. **6語制限が短すぎる**。実用的なビジネス英語が入らず、汎用句になりがち。
4. ASR に言語ヒントも直前文脈も渡していないため、日英混在の発話で精度が落ちている。

**解決策**:
1. **日本語断片を第一の意図ソースにする**。プロンプトに明示: 「学習者が日本語を発した場合、それが言いたいことの最も確実な手がかりである。文脈から推測するのではなく、その日本語の意図を口に出せる英語に変換せよ」。ASR は言語を固定せず日英混在を許容する。
2. **継続モードを既定にする**。学習者が文を言いかけている場合、choices は**その続きとして自然につながる文**であること（学習者は直前に言った内容の後に、提示された文をそのまま発話する）。言いかけていない場合のみ独立した文を出す。判定結果に `continuation: true|false` を持たせ、HUD では継続時に "…" を先頭に付けて視覚的に示す。
3. **語数上限を6語から10語に緩和**（HUD 2行以内なら可読）。ただし冗長な前置き（"I was wondering if you could possibly..."）は禁止し、口語で短く。
4. **ASR にコンテキストを渡す**: transcription API の prompt パラメータに直近の会話文字列を渡し、固有名詞・専門語・日英混在の認識精度を上げる。
5. **WORD カードの実用性向上**: 難単語の言い換えだけでなく、その語を使った短い例文が有用な場合は例文を優先する。
